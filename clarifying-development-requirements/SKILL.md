# Clarifying Development Requirements

## Overview

把零散想法逐项澄清为严谨、可执行、可验收的开发需求。先确认真实意图，再整理文字；完整看起来合理但未经确认的需求，仍然是不可靠的需求。

## 核心流程

严格按以下阶段工作：

1. **识别任务类型**：只分析、缺陷修复、功能开发、重构、UI 调整，或其他开发任务。
2. **提取现有信息**：区分“已确认”“待确认”“实施时决定”。
3. **检查适用字段**：只检查当前任务需要的字段，不机械套用全部模板。
4. **选择一个问题**：从待确认项中只问当前影响最大的一个问题。
5. **逐轮澄清**：收到回答后更新信息状态，再选择下一个问题。
6. **检查完成门槛**：关键未知项未解决时，继续提问。
7. **生成最终需求**：门槛满足后，按固定结构输出。

如果用户同时要求直接实现，先完成澄清并让用户确认最终需求，再进行任何修改。

## 信息状态

| 状态 | 判定 |
|---|---|
| 已确认 | 用户明确说过，或来自用户指定且已经读取的代码、日志、截图、文档 |
| 待确认 | 不同答案会改变业务结果、权限、数据兼容、数据库结构、公共接口、修改范围或验收结论 |
| 实施时决定 | 不改变外部行为，可依据现有架构和项目规范选择的内部技术细节 |

不要把常见做法、合理猜测或推荐方案写成“已确认”。需要帮助用户表达时，可以在同一个问题中给出 2–3 个候选解释，请用户选择或修正。

## 每轮提问契约

每次回复只包含：

1. 一句简短的当前理解；
2. 一个最高优先级问题；
3. 必要时附带 2–3 个互斥候选答案。

不要同时提出第二个问题，不要附带完整需求草稿。

按以下优先级选择问题：

1. 交付方式：只整理或分析，还是确认后直接修改；
2. 目标与期望结果；
3. 会改变业务语义、权限或数据结果的规则；
4. 修改范围、不能改变的内容和兼容要求；
5. 验收标准；
6. 定位、复现和环境信息；
7. 分支、worktree、提交和合并方式。

如果用户的回答仍有多种实现含义，继续围绕同一问题澄清，不要跳到下一项。

## 通用字段

最终需求必须覆盖或明确标记“不适用”：

- 任务类型；
- 目标；
- 相关位置，或需要先定位的现象与入口；
- 当前情况；
- 期望结果；
- 业务规则与关键边界；
- 修改范围与非目标；
- 验收标准；
- 执行方式。

未知文件或方法名可以写“需要先定位”，但必须有足够的现象、入口、错误或操作线索用于定位。

## 按任务类型检查

| 类型 | 额外必填信息 |
|---|---|
| 缺陷修复 | 复现步骤、实际结果、期望结果、运行环境、复现稳定性、可用日志或截图、不能改变的行为 |
| 功能开发 | 使用者与权限、主流程、数据规则、适用的空值/重复/无权限/异常/部分成功/并发边界、允许修改范围、非目标 |
| 重构 | 当前结构问题、必须保持的外部行为、允许修改的内部实现、禁止范围、行为等价验证 |
| UI 调整 | 页面或截图、具体区域、可观察问题、可判断目标、适用交互状态、响应式尺寸、不允许影响的样式 |
| 只分析 | 分析对象、期望输出、明确禁止修改文件和外部状态 |

只询问确实适用的边界。例如功能没有并发或部分成功语义时，将其标记为不适用，不制造无意义追问。

## 完成门槛

仅当以下条件全部满足时生成最终需求：

- 目标和期望结果可以客观判断；
- 关键业务规则、权限、数据与兼容边界已确认或明确不适用；
- 修改范围和非目标不会产生重大歧义；
- 验收方式足以证明完成；
- 允许执行的动作和外部变更范围明确。

用户可以把不改变业务语义的内部实现委托给现有项目规范。用户不能用“按最佳实践处理”代替业务规则、权限、数据兼容、数据库结构或公共接口的确认。

## 最终输出契约

门槛满足后，固定输出：

### 需求理解确认

用 2–4 句话复述目标、关键边界和执行方式。

### 整理后的开发需求

按适用模板写出可以直接交给开发代理的完整需求，使用明确、可观察、可验证的语言。

### 实施时可自行决定

只列不影响业务语义的内部技术选择；没有则写“无”。

### 未覆盖内容

列出明确排除或暂不处理的内容；没有则写“无”。

最后请用户确认整理结果。用户确认前，不执行代码或外部状态修改。

## 示例

用户：

> 我想在后台加一个导出按钮，把聊天记录导出来。

第一轮只问：

> 我理解你希望新增后台聊天记录导出能力。这个功能确认需求后是只输出开发说明，还是需要开发代理直接实现？

用户回答“直接实现”后，下一轮只问：

> 明白，最终需要直接实现。哪些角色可以导出，以及每个角色允许导出哪些范围的数据？

继续逐轮确认导出范围、筛选语义、格式与字段、空结果和失败处理、修改限制、验收方式。不要提前假定管理员、Excel、异步任务、审计日志或脱敏规则。

## 快速检查

在输出最终需求前确认：

- 当前是在提问阶段，还是已经满足完成门槛？
- 本轮是否只有一个问题？
- 是否把猜测写成了事实？
- 是否遗漏权限、数据、异常、兼容或范围边界？
- 验收标准能否判断成功或失败？
- “只分析”和“直接修改”的权限是否清楚？

## 常见错误

| 错误 | 修正 |
|---|---|
| 为了显得专业而补出角色、格式、字段或技术方案 | 把它们作为候选答案逐项确认 |
| 一次发送整张问卷 | 只问最高优先级的一个问题 |
| 信息不足时先输出“初版需求” | 保持提问阶段，满足完成门槛后再输出 |
| 所有任务都询问全部异常边界 | 先判断是否适用，再确认 |
| 因未知内部实现而持续阻塞 | 将不影响外部行为的细节归入“实施时决定” |
| 用户说“可以”就视为批准所有事项 | 明确复述被批准的范围、限制和执行方式 |
